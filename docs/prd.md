## 개요

PDF 단어장(일본어–한국어)을 업로드하면 AI(Gemini 2.5 Flash)가 단어를 구조화하고 유의어·예문을 붙여, 4지선다 퀴즈와 경량 SRS로 혼자 일본어 단어를 외우는 **1인 개인용** 학습 웹앱.

- 기술 스택(확정): Next.js(App Router, TS) · Prisma + **로컬 SQLite** · Gemini API(**2.5 Flash**) · unpdf · Tailwind.
- 인증: 무인증(`userId=1` 고정). 배포: 로드맵 W4 선택사항(그때 결정).
- 범위: MVP(P0) — 업로드→파싱→검수→생성→단어장 뷰→양방향 4지선다→경량 SRS→진도 대시보드.
- **개정(2026-07-06)**: MVP(#1~#9) 완료 후 사용성 피드백을 반영해 **UI/UX 개선(#10)** 추가 — 글로벌 네비게이션, 퀴즈 인터랙션(피드백 단계·문제 수 선택·힌트·정답 보기), 로딩 표시, 모던 디자인(`design-system.md`). 상세는 `spec-fixed.md` R5 개정·R10.
- **개정(2026-07-06, 2차)**: **퀴즈 학습 옵션 확장(#11)** — 퀴즈 유형 3종(한자→뜻 / 후리가나→뜻 / 뜻→단어), 출제 방식 4종(전체 랜덤 / 정답 제외 / 복습 우선 / 저장 단어만), 단어 저장(북마크). 상세는 `spec-fixed.md` R4·R5 개정.
- **개정(2026-07-06, 3차)**: **단어장 관리·내비게이션 개선(#12~#14)** — 하단 고정 내비게이션(홈/뒤로가기/앞으로가기), 단어장 이름 수정·삭제, 마지막 학습 날짜 표시. 상세는 `spec-fixed.md` R4·R10 3차 개정.

## 사용자 스토리

1. 학습자로서, 내 PDF 단어장을 업로드하면 단어가 `漢字/후리가나/뜻`으로 구조화되어 보이길 원한다. (R1)
2. 학습자로서, 잘못 파싱된 행을 검수 화면에서 고치고 "확정"하면 그때 AI 생성이 시작되길 원한다. (R2, R3)
3. 학습자로서, 각 단어에 유의어·예문이 붙어 실사용 감각을 익히길 원한다. (R3)
4. 학습자로서, 매일 "오늘 복습할 단어"를 양방향 4지선다로 풀고, 정·오답에 따라 다음 복습일이 자동 조정되길 원한다. (R5, R6)
5. 학습자로서, 대시보드에서 진도율과 오늘 복습 수를 한눈에 보길 원한다. (R7)
6. 학습자로서, 어느 화면에서든 홈으로 돌아가고 언제든 새 단어장을 만들 수 있길 원한다. (R10)
7. 학습자로서, 홈 목록에서 단어장을 클릭해 상세로 바로 들어가고, 기다리는 동안 로딩 상태를 보길 원한다. (R10)
8. 학습자로서, 퀴즈에서 답을 고르면 정답/오답 피드백을 확인한 뒤 내 속도로 다음 문제로 넘어가고, 시작 전에 문제 수를 고르고, 모르는 문제는 힌트나 정답을 볼 수 있길 원한다. (R5 개정)
9. 학습자로서, 퀴즈 유형(한자→뜻 / 후리가나→뜻 / 뜻→단어)을 골라 한자와 읽기를 나눠 연습하길 원한다. (R5 개정)
10. 학습자로서, 출제 방식(전체 / 이미 맞힌 단어 제외 / 복습 우선 / 내가 저장한 단어만)을 골라 지금 학습 상황에 맞게 퀴즈를 돌리길 원한다. (R5 개정)
11. 학습자로서, 단어장에서 외우고 싶은 단어를 '저장'해 두고 그 단어들만 따로 퀴즈로 풀길 원한다. (R4 개정)
12. 학습자로서, 어느 화면에서든 하단의 홈/뒤로가기/앞으로가기 버튼으로 빠르게 이동하길 원한다. (R10 3차 개정)
13. 학습자로서, 단어장 이름을 잘못 지었을 때 바로 고치고, 더는 안 쓰는 단어장은 지우길 원한다. (R4 3차 개정)
14. 학습자로서, 각 단어장을 마지막으로 언제 공부했는지 한눈에 보길 원한다. (R4 3차 개정)

## 기술 결정 (ADR)

### ADR-1. AI 잡 실행: 서버 잡 + 상태 폴링 (2안)

**Context** — 파싱(Pass 1)·생성(Pass 2)은 576단어 기준 수십 회의 청크 호출이 필요한 오래 걸리는 작업이다. 진행률을 사용자에게 보여줘야 하고, 새로고침·이탈 후 재방문 시 진행 상태가 이어져야 한다. 진행률 전달 방식은 "폴링"으로 확정된 상태다.

**Decision** — 잡 실행을 **서버가 소유**한다. 잡 시작 요청 시 서버가 잡 레코드를 만들고, 청크 루프를 돌며 각 청크 처리 후 진행률(`processed/total`)과 상태를 DB에 갱신한다. 클라이언트는 `GET /api/jobs/:id`를 1~2초 간격으로 폴링해 진행바를 그린다. 청크 처리는 **순수 함수**(입력 텍스트/단어 배열 → 결과 배열)로 분리하고, 잡 러너가 이를 호출·집계·재시도(청크당 최대 3회)한다.

**Alternatives**
- *1안 클라이언트 오케스트레이션* — 거부. 진행 상태가 클라에만 있어 새로고침·이탈 시 유실되고, 오케스트레이션 로직이 UI에 흩어져 테스트가 어렵다.
- *3안 큐/워커 분리(BullMQ/Redis)* — 거부. 확장성은 좋지만 1인 로컬 개인용 MVP에 Redis·워커 프로세스 인프라는 과하고 셋업·테스트 비용이 크다.

**Consequences**
- (+) 청크 순수 함수 + 잡 러너 분리로 단위·통합 테스트가 쉽다(이슈 단위 TDD 적합).
- (+) 진행 상태가 DB에 있어 이탈/새로고침에 강하다.
- (−) Vercel 서버리스 배포 시 함수 실행 시간 제한과 충돌할 수 있다. → 지금은 로컬 SQLite·로컬 실행이라 무관하며, 배포(W4)에서 롱러닝 처리 방식을 별도 결정한다(Out of Scope).
- (−) 폴링은 SSE보다 약간의 지연·중복 요청이 있으나 개인용 규모에서 무시 가능.

### ADR-2. DB: 로컬 SQLite 파일 (Prisma)

**Context** — 1인 개인용이고 배포는 W4의 선택사항이다.
**Decision** — `file:./dev.db` SQLite + Prisma로 시작. `userId`는 기본값 1로 채우되 컬럼은 유지.
**Alternatives** — Turso/Vercel Postgres 거부: 서버리스 휘발 대비 이점은 있으나 배포 확정 전이라 초기 셋업 복잡도만 늘린다.
**Consequences** — (+) 운영부담 0, 마이그레이션 간편. (−) Vercel 배포 시 파일 휘발 → 배포 시 Turso 등으로 교체 필요(W4에서 결정).

### ADR-3. AI 모델: Gemini 2.5 Flash 단일 (개정 — 원래 Claude Haiku 4.5)

**Context** — 파싱·생성 두 곳에서 LLM을 호출하며 개인용 비용에 민감. 개발자가 Anthropic 대신 Google AI Studio 키(무료 티어)를 사용하기로 함.
**Decision** — 두 Pass 모두 `gemini-2.5-flash` (`@google/genai` SDK). 키는 `GEMINI_API_KEY` 환경변수. 구조화 출력은 Gemini 네이티브 JSON 모드(`responseMimeType: "application/json"`)를 사용.
**Alternatives**
- *Claude Haiku 4.5(원안)* — 거부. 개발자가 Anthropic 키가 없고 Gemini 무료 티어를 선호.
- *gemini-2.5-pro / 2.5-flash-lite* — 거부. pro는 반복 호출에 과함, lite는 한컴 PDF 파싱 정확도 우려.
**Consequences** — (+) 무료 티어로 사실상 무비용, 고속. (+) 네이티브 JSON 모드로 "JSON만 출력" 신뢰성↑ → 펜스 제거 로직 부담 감소(단, 방어적 파싱은 유지). (−) 한컴 PDF 파싱 오류 가능성은 여전 → **검수 화면 강화**로 상쇄. 파싱 정확도가 부족하면 Pass 1만 pro로 올리는 것을 재검토.

### ADR-4. 생성 타이밍: 검수 확정 후

**Context** — 원본 명세가 "업로드 즉시"와 "검수 후"로 상충.
**Decision** — 사용자가 검수 화면에서 확정한 단어에 대해서만 Enrichment(Pass 2) 시작.
**Alternatives** — 업로드 즉시 생성 거부: 삭제될 행에 비용이 낭비되고 재생성 로직이 복잡.
**Consequences** — (+) 비용 절약, 파이프라인 단순. (−) 확정 전에는 유의어·예문이 없어 사용자가 잠시 기다림(진행바로 커버).

### ADR-5. 마지막 학습 날짜: 조회 시점 집계 (QuizAttempt 재사용)

**Context** — 단어장별 "마지막 학습 날짜"를 보여줘야 한다. 이미 `QuizAttempt.answeredAt`에 모든 답안 시각이 기록되고 있다.
**Decision** — 별도 컬럼 없이, 단어장 상세/홈 목록 조회 시점에 그 단어장 소속 단어들의 `QuizAttempt.answeredAt` 중 최댓값을 집계해 보여준다.
**Alternatives**
- *`VocabSet.lastStudiedAt` 컬럼 추가 + 퀴즈 답안 제출마다 갱신* — 거부. 조회는 빨라지지만 마이그레이션과 갱신 로직(quiz-attempts API에 side-effect 추가)이 필요하고, 이미 기록 중인 데이터의 중복 저장이라 1인 개인용 규모에서 이득이 적다.
**Consequences** — (+) 스키마 변경 없음, 데이터 정합성 문제(갱신 누락) 원천 차단. (−) 단어장 개수·QuizAttempt 양이 매우 커지면 집계 쿼리 비용 증가 — 1인 개인용 로컬 SQLite 규모에서는 무시 가능, 필요 시 나중에 컬럼화 재검토.

## Out of Scope

이번 MVP에서 **하지 않는다**(명시하지 않으면 구현 범위가 의도치 않게 확장됨):

- 주관식 입력 퀴즈, 예문 빈칸 채우기 퀴즈 (P1)
- AI 생성 예문·유의어 **인라인 수정** (P1) — 이번엔 생성물 저장까지만
- TTS 발음 듣기 (P2)
- 후리가나 맞추기 퀴즈, 반의어·JLPT 레벨·품사 (P2)
- 멀티유저·로그인 — `userId` 컬럼만 예약, 실제 인증/세션 없음
- 상세 통계 그래프, 연속 학습(스트릭)
- **배포 및 서버리스 DB 전환(Turso/Postgres)** — 로드맵 W4로 연기. MVP는 로컬 실행 기준
- 다중 파일 동시 업로드 (1회 1파일)
- 실시간 SSE/WebSocket 진행률 (폴링만)
- 단어장 공유·내보내기(export)
- 이미지/스캔 PDF의 OCR (텍스트 추출 가능한 PDF만)
- **단어장 삭제 되돌리기(휴지통/복구)** (추가 2026-07-06 3차) — `confirm()` 확인 후 영구 삭제. 실수 복구는 범위 밖
- **커스텀 내비게이션 히스토리 스택** (추가 2026-07-06 3차) — 뒤로/앞으로가기는 브라우저 세션 히스토리 그대로 사용. 앱 자체 히스토리 관리 없음
- **단어장 이름 중복 검사** (추가 2026-07-06 3차) — 동일 이름 여러 단어장 허용(기존 확정 흐름과 동일 정책)
