# TDD Green

현재 **실패 중인 테스트를 통과시키는 최소한의 구현 코드만** 작성하는 단계.
테스트가 요구하지 않는 기능은 만들지 않는다. 테스트가 통과하면 그 항목은 끝이다.

**입력**: 이슈 번호 (`$ARGUMENTS`)

이슈 번호로 시나리오 문서 `docs/features/{feature}/issue-{N}.md`를 찾는다.
`{feature}`를 모르면 `docs/features/` 아래에서 `issue-{N}.md`를 검색해 확정하고,
찾지 못하면 사용자에게 피처 이름을 물어본다.

## 핵심 원칙

이 단계의 유일한 목표는 **빨간 테스트를 초록으로 바꾸는 것**이다. 그 이상도 이하도 아니다.
"나중에 필요할 것 같은" 코드, 테스트가 검증하지 않는 분기, 미리 만든 추상화는
전부 Green 단계의 범위를 벗어난다. 아름다움은 다음 Refactor 단계에서 다룬다.

## 제약 (반드시 지킨다)

- **테스트에 없는 기능은 구현하지 않는다.** 테스트가 요구하는 동작만 만든다.
- **테스트 파일은 수정하지 않는다.** 테스트가 실패하면 구현을 고치지, 테스트를 고치지 않는다.
- **불필요한 추상화를 만들지 않는다.** 지금 한 곳에서만 쓰이면 그냥 인라인으로 둔다.
- **design.md / design-system.md에 없는 시각적 요소를 임의로 추가하지 않는다.**
  색상·간격·아이콘 등은 디자인 문서에 근거가 있을 때만 넣는다.

이 제약들은 Green 단계가 "최소 구현"이라는 정체성을 지키기 위한 것이다.
범위를 넘으면 피드백 루프가 흐려지고, 어떤 코드가 어떤 테스트 때문에 존재하는지
추적할 수 없게 된다.

## 실행 순서

### 1. 실패 테스트 목록 확인

```bash
npm test
```

vitest가 보고하는 실패 테스트 목록을 확보한다. 이후 작업의 기준점이 된다.

### 2. collect 실패 감지 (모듈 stub 생성)

vitest는 `import` 대상 모듈이 없으면 해당 테스트 파일 전체를 **수집(collect) 단계에서**
실패시켜, 실제 테스트 케이스 수보다 적게 보고한다. 이 상태에서는 무엇을 통과시켜야 하는지
정확히 알 수 없다.

**감지 방법**: 시나리오 문서 `docs/features/{feature}/issue-{N}.md`의 체크박스(`- [ ]`)
**총 건수**와 vitest가 보고하는 `Tests` 수를 비교한다.

- **일치** → 모든 테스트가 인식되고 있다. 3번으로 넘어간다.
- **불일치** → import 대상 모듈이 없어 collect가 깨진 것이다. 실패한 import 경로를 읽고,
  **시그니처만 있는 stub 파일**을 생성해 전체 테스트가 인식되게 한다.

stub은 이름·인자·반환 타입만 갖춘 빈 껍데기다. 로직을 넣지 않는다.
목적은 오직 "vitest가 모든 테스트 케이스를 수집하게" 만드는 것이다. 예:

```ts
// stub — 시그니처만. 로직은 4번 단계에서 테스트를 보며 채운다.
export function calculateScore(answers: Answer[]): number {
  throw new Error('not implemented')
}
```

stub 생성 후 `npm test`를 다시 실행해 `Tests` 수와 체크박스 수가 일치하는지 확인한다.

### 3. 디자인 컨텍스트 로드

구현 대상이 UI 컴포넌트라면 시각적 스펙을 먼저 파악한다. 순서:

1. `docs/design-system.md`가 있으면 읽고, 구현 대상 컴포넌트의 **색상·간격·레이아웃·
   상태별 표현**(hover, disabled, error 등)을 확인한다.
2. `docs/features/{feature}/issue-{N}.md`에 **"디자인 참고"** 섹션이 있으면 함께 읽는다.
3. `CLAUDE.md`의 스타일링 컨벤션을 참조한다.

**디자인 문서가 하나도 없으면 이 단계를 건너뛰고 `CLAUDE.md` 컨벤션만 따른다.**
UI가 아닌 로직 구현이면 이 단계는 생략한다.

### 4. 첫 번째 실패 테스트를 통과시키는 최소 코드 작성

실패 목록의 **맨 위 테스트 하나**만 겨냥한다. 그 테스트가 검증하는 동작을 충족하는
가장 단순한 코드를 작성한다.

- 여러 테스트를 한꺼번에 통과시키려 하지 않는다. 하나씩 초록으로 만든다.
- UI 컴포넌트라면 테스트가 요구하는 동작을 충족하되, 3번에서 파악한 디자인 스펙을
  **함께** 적용한다. (동작은 테스트가, 외형은 디자인 문서가 근거다.)

### 5. 재실행 + 회귀 확인 (피드백 루프)

```bash
npm test
```

겨냥한 테스트가 통과하고 **기존 테스트가 깨지지 않았는지**(회귀 없음) 전체를 확인한다.

- **여전히 실패** → 에러 메시지를 읽고 원인을 분석해 코드를 수정한 뒤 재실행한다.
  이 수정→재실행 루프는 **최대 5회**까지 반복한다.
- **5회 후에도 실패** → 임의로 계속 시도하지 않는다. 실패 원인(에러 메시지, 시도한 접근,
  막힌 지점)을 개발자에게 **보고하고 중단**한다.

### 6. 시나리오 문서 체크

통과한 테스트에 해당하는 `docs/features/{feature}/issue-{N}.md`의 항목을 `- [x]`로 체크한다.
이 체크박스가 남은 작업량의 단일 기준이 된다.

### 7. 다음 실패 테스트로 이동

남은 실패 테스트가 있으면 **4번으로 돌아가** 다음 하나를 겨냥한다.
모든 테스트가 통과할 때까지 4→5→6→7을 반복한다.

### 8. 전체 통과 후 커버리지 측정 (참고용)

```bash
npx vitest run --coverage
```

미커버 라인을 확인하고, 그것이 **시나리오에서 빠진 케이스인지** 판단해 개발자에게 보고한다.

- **커버리지를 올리기 위한 테스트 추가는 금지.** 커버리지는 시나리오의 누락을 발견하는
  참고 지표로만 쓴다. 테스트를 만드는 것은 Red 단계의 몫이다.
- 미커버 라인이 "테스트가 없어서"라면, 그 시나리오를 개발자에게 알려 Red 단계에서
  다룰지 결정하게 한다.

## 한눈에 보는 루프

```
npm test (1) → collect 확인·stub (2) → 디자인 로드 (3)
        ↓
   ┌─→ 실패 테스트 1개 겨냥 (4)
   │        ↓
   │   npm test (5) ── 실패 ──→ 원인 분석·수정 (최대 5회) ──→ 5회 초과 시 보고·중단
   │        │ 통과
   │        ↓
   │   issue-{N}.md 체크 (6)
   │        ↓
   └── 남은 실패 있음? (7) ── 예
            │ 아니오 (전체 통과)
            ↓
     커버리지 측정·보고 (8, 참고용)
```
